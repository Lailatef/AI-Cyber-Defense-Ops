# Investigation: CA_DCSync_4662.evtx

## Summary

Hayabusa scanned 3 Security-channel events from `DC1.insecurebank.local` and
flagged all 3 as critical matches for the `dcsync` Sigma rule ("Active
Directory Replication from Non-Machine Account (DCSync)", RuleID
`5d2e8b3a-9c1f-4e6d-a7b2-3f4c5d6e7a8b`), mapping to [[T1003.006]]. Every
event is an EventID 4662 (object access) fired at the exact same timestamp
(2019-05-07 22:10:43.487 -04:00) with sequential Record IDs
(202791-202793), which is consistent with a single automated tool call
(e.g. Mimikatz `lsadump::dcsync` or Impacket `secretsdump.py`) rather than
three independent manual actions â€” this kind of sub-second, multi-event
burst is a strong timing-cluster indicator.

The subject performing the access is `insecurebank\Administrator`, with
`SubjectUserSid` ending in `-500` (the well-known RID for the built-in
domain Administrator account). The `Properties` field across the three
events references both the `DS-Replication-Get-Changes`
(`1131f6aa-9c07-11d1-f79f-00c04fc2dcd2`) and `DS-Replication-Get-Changes-All`
(`1131f6ad-9c07-11d1-f79f-00c04fc2dcd2`) extended rights against a
`domainDNS` object â€” the canonical DCSync right pair. Legitimate AD
replication is normally performed by domain controller machine accounts
(accounts ending in `$`), which the Sigma rule's `filter_dc_accounts`
condition explicitly excludes; the fact that a named user account
(`Administrator`) rather than a machine account triggered this is the core
suspicious signal and the reason the rule fired at all. All 3 hits share the
same `RuleTitle`/`RuleID`, so there is no additional rule diversity to
correlate against in this scan.

Re-running the scan against the current state of `rule_dir`
(`Module-5/mcp-detection-kb/rules/`, 16 `.yml` files) shows all 16 rules
load and parse cleanly under `hayabusa -v` â€” zero `[WARN]`/parsing errors.
This is an improvement over the prior run of this same investigation earlier
today: `lsass_minidump.yml`'s duplicate `CommandLine|contains` key (which
previously made it fail to load) is no longer present in the file, and
`brute_force.yml`'s aggregation-condition issue was already fixed by
splitting it into `brute_force.yml` + `brute_force_single_account.yml`. Both
fixes are reflected in this run's coverage table below with no caveats
remaining.
